# 建站笔记：我踩过的那些建站平台（平台选型心得）

> 从 2015 年起，我就开始尝试搭建个人网站。时至今日，在经历了各种平台的变迁之后，我最终在
> [`wiki.hanvon.top`](https://wiki.hanvon.top/) 上搭建了个人知识库，在这里记录与分享知识。
>
> 这篇把我「为什么自建知识库 / 为什么公开 / 为什么不用第三方平台 / 怎么选型」想清楚，并把**我用过或调研过的几乎所有建站平台**列出来、逐一点评，给同样在折腾站点的人一个参考。
>
> 本文改编自 Power's Wiki《[为什么你需要一个知识库](https://wiki-power.com/)》一文（**CC BY-NC-SA 4.0**），结合我的实际建站经历重写，转载请注明出处。

---

## 一、为什么我要自建知识库？

**知识需要归档沉淀**。我平时学习积累的知识，如果不及时整理归纳，很快就会忘记。合理的分类归档，既有利于日后检索与参考，又能避免在同一个地方踩坑。

**用输出倒逼输入**。若想快速掌握一门知识，最好的方法就是用自己的话复述给别人听。当我能把知识清晰地描述出来，才算真正掌握它。持续输出会反过来逼着我去阅读、去补全薄弱环节。

**赛博空间的桃源**。在这一片自留地上，我可以避开主流互联网上的纷争，静下心来耕耘。我把个人知识库当作世外桃源一样的存在。

---

## 二、为什么知识库要公开？

- **更精致化的输出和整理**：不公开的话要求就变低了，归纳整理容易慢慢随意，久了变成草稿箱，甚至看不懂自己先前写了什么。
- **分享让思想碰撞**：知识需要分享，在碰撞中可能出现新的火花；别人遇到一样的问题，直接发我写过的教程，既方便又酷。
- **提升个人品牌，遇见志同道合的人**：持续维护一个知识库，会收获同好的读者与朋友；把链接挂在社交平台首页或简历上，也是实打实的个人实力展示。

---

## 三、为什么自建，而不是用第三方平台？

第三方平台指的是**语雀、知乎、简书、公众号、CSDN** 这一类。它们帮你搞定底层构建、图片存放、部署与 SEO，你只管写。既然这么省事，为什么我还提倡自建？

首先，是 **数据安全**。你在第三方平台留下的文字，所有权并不归属你。大多数平台都在筑墙：接受数据导入，却给批量迁出设障碍；很多还阻止外链，不利于知识交流。把数据完全托付给平台，是非常不保险的——封禁、删改、服务器被攻击或平台倒闭，你都没有主动权。

其次，是 **可定制性**。选了平台，就得接受它的样式和 UI，有时还得忍受边角的牛皮藓广告。自建则 UI 完全自定义，清爽简洁的阅读体验提升巨大。

说到底，第三方与自建的关系，就像**租房子与买房子**：租房子入住方便、付出少，但拥有的是使用权不是所有权；买来的房子，装修风格、堆放什么都随我。

---

## 四、知识库，而不是文章流？

「文章流」指博客、公众号及大部分平台：以**时间线**归档文章。而「知识库（Wiki）」以**类别**收纳文章，像翻阅档案，更利于检索和分类阅读。

一些琐碎但不可或缺的知识，并不适合单独成文；体系化的文章，又是从这些零散知识生长而来。以类别做区分，正好合适。卡片式写作也是同理：基础知识是一张张小卡片，顶层文章调用底层卡片，提高复用、减少重复。

## 五、我的选型标准

知识库平台，我看重这几点：

- **基础功能**：侧边栏显示标题与类别，主版面显示内容，支持 Markdown，内容与框架相互独立。
- **搜索功能**：类别归档之外，全站搜索能直接定位知识点。
- **项目开源**：闭源项目可能因维护者原因停更（此处特指 GitBook、Bitcron），遇到 bug 也难以及时解决。
- **静态网站**：能托管在 GitHub Pages 等平台，不用单独买服务器，省钱易维护，对 SEO 友好。
- **易搭建、易部署更新**：架设不该花过多时间，更新流程不应过于复杂。
- **UI 简洁美观**：为阅读体验，版面尽量清爽，最好有夜间模式。

## 六、自建知识库的局限（平衡视角）

- **流量问题**：第三方平台更容易被读到、更容易上搜索前十。自建站一般流量很小、读者不多。我的解法是「一文多发」（用 Artipub 等工具同步到各平台，把流量导回个人站）。
- **有一定技术门槛**：Jamstack（JS + API + Markup）理念及相关技术链，需要一点学习。不过一旦跑通，日常就是写 Markdown + push。
- **静态站无在线编辑器**：编辑器和网站分离，无法像知乎那样网页/App 直接编辑。我用一个本地编辑器（VS Code / Typora / Obsidian）解决；配合 GitHub Actions，push 即上线。
- **知识点可隶属多类别**：知识库一般单篇只归一个类别。遇到交叉内容（如既属 `电路设计` 又属 `STM32`），我按相关度更高的类别收录——不用标签系统，避免标签泛滥后检索反而不如类别 + 搜索。

## 七、我都用过哪些平台？

下面是我亲身用过或认真调研过的平台，逐一点评（评价口径沿用并改编自 Power's Wiki 的选型笔记）：

| 平台 | 类型 | 我的点评 |
|---|---|---|
| **WordPress** | 动态 / 数据库 | 使用方便，但底层过于庞大复杂，需要数据库，不利于迁出。 |
| **Hexo** | 静态 / 博客 | 较为冗杂，且没有比较好看的 Wiki 主题（做博客还行）。 |
| **GitBook** | 静态 / 文档 | CLI 版本已停止更新支持，V2 版本国内访问速度较慢。 |
| **Jekyll** | 静态 / 博客 | 技术相对较旧，且缺少更新支持。 |
| **GitHub Issues / Gist / Wiki** | 平台内 | 国内访问速度较慢，UI 不可定制。 |
| **Bitcron** | 动态托管 | 可定制性较差，访问速度时快时慢。 |
| **DokuWiki** | 动态 / 需服务器 | 需自备服务器，且本身过于老旧。 |
| **Gridea** | 静态 | 部署简单，但可定制性较差，只能用其专用编辑器。 |
| **wiki.js** | 动态 / 需服务器 | 需自备服务器搭建，一般适用于团队知识库，存在部分小问题。 |
| **语雀** | 第三方知识库 | 相对不错的第三方平台，缺点是 UI 不可定制，且数据迁出不方便。 |
| **Hugo** | 静态 | 部署速度快，但没有比较好看的 Wiki 主题（做导航 / 作品集很香）。 |
| **Gatsby** | 静态 | 与 Hugo 相似，偏重、生态偏 React。 |
| **Ghost** | 动态 | UI 美观，需要自备服务器（或花钱买服务），可定制性较差。 |
| **docsify** | 静态（加载渲染） | 比较推荐。部署简易，UI 美观，但加载时渲染，某些设备性能较差。 |
| **Docute** | 静态（加载渲染） | 比较推荐。部署简易，UI 美观，比 docsify 插件更少。 |
| **VuePress** | 静态 | 比较推荐。各方面不错，社区插件多；局限是官方文档较乱、有些小 bug。 |
| **Docusaurus** | 静态 | 比较推荐。各方面不错；局限是编译较慢、框架较臃肿。 |
| **MkDocs（Material 主题）** | 静态 | 比较推荐。各方面不错，编译也快——**我现在的方案**。 |

> 第三方平台再提一嘴：语雀、知乎、简书、公众号、CSDN 这类「不用管底层」的平台，适合纯写作分发的场景；但只要你在意**数据所有权和可定制性**，迟早会回到自建。

## 八、我现在用什么？

踩了一圈坑之后，我的分工是**按内容模型选引擎**（详见 [《博客引擎的选择》→](建站笔记-博客引擎的选择.md)）：

- **wiki（知识库）+ digest（书摘）= MkDocs Material**：一页一篇 Markdown，Obsidian `[[wikilink]]` 直转，搜索/分类现成，编译快。
- **nav（导航）+ works（作品集）= Hugo**：内容是一组「数据」而非正文，用 `data/*.yaml` 驱动 + 自定义布局，对齐参考站观感（见 [Hugo 对照笔记 →](建站笔记-Hugo对照.md) 与 [从零搭 Hugo →](建站笔记-Hugo搭建.md)）。
- **用户页博客（Hexo）= `wild-civil.github.io` → `blog2.hanvon.top`**：经典时间线博客，主题生态成熟、最省心。

也就是说：**文档交给 MkDocs，数据/观感交给 Hugo，博客流交给 Hexo**。没有万能引擎，按内容拆站反而每个站都趁手。

---

## 九、小结

临渊羡鱼，不如退而结网。与其纠结「哪个平台最好」，不如先想清楚：**我的内容到底是文档、是数据、还是时间线文章？** 想明白这点，平台自然就浮现了。

对我而言，答案已经稳定：知识库与书摘用 MkDocs，导航与作品集用 Hugo，博客用 Hexo——三者都托管在 GitHub Pages，push 即上线。
